REPLICATION PROTOCOL SPECIFICATION
CS 464/564 Assignment 4 – Two-Phase Commit (2PC) Application Protocol
=======================================================================

1. OVERVIEW
-----------
The replication protocol is a plain-text, line-oriented TCP protocol used
between replica servers (peers) to keep their bulletin-board files
identical.  Each line is terminated by a single newline character ('\n').
Carriage returns ('\r') are silently ignored, so telnet-style CRLF lines
are also accepted.

A server that receives a WRITE or REPLACE command from a client becomes the
MASTER (coordinator) for that transaction.  All other servers listed in its
PEER configuration are SLAVES (participants).  The master opens one TCP
connection to each slave on that slave's RPORT and runs the two-phase commit
protocol over those connections.

READ commands are served locally; they never involve the replication
protocol.

2. PORT
-------
Each server listens for incoming 2PC connections on its replica port
(RPORT, default 9001).  Client traffic uses the separate BBPORT.

3. MESSAGE SYNTAX
-----------------
All messages begin with the literal prefix "2PC " followed by a keyword and
optional arguments.  Arguments are separated by single space characters.
Message format (EBNF):

  message          ::= direction SP* "\n"
  direction        ::= master_msg | slave_msg

  master_msg       ::= PRECOMMIT
                     | ABORT
                     | COMMIT SP operation
                     | SUCCESS
                     | UNSUCCESS

  slave_msg        ::= ACK SP ack_result
                     | DONE SP done_result

  operation        ::= WRITE SP poster SP message_text
                     | REPLACE SP msg_id SP poster SP message_text

  ack_result       ::= "OK" | "FAIL"
  done_result      ::= "OK" | "FAIL"

  poster           ::= one token (no spaces, no '/')
  message_text     ::= remainder of line (may contain spaces, no '\n')
  msg_id           ::= positive integer

Full wire representations:

  "2PC PRECOMMIT\n"
  "2PC ABORT\n"
  "2PC COMMIT WRITE <poster> <message_text>\n"
  "2PC COMMIT REPLACE <msg_id> <poster> <message_text>\n"
  "2PC SUCCESS\n"
  "2PC UNSUCCESS\n"
  "2PC ACK OK\n"
  "2PC ACK FAIL\n"
  "2PC DONE OK\n"
  "2PC DONE FAIL\n"

4. PROTOCOL SEQUENCE (normal path – all slaves available)
----------------------------------------------------------

TIME →          MASTER                          SLAVE
                  |                               |
  client sends    |                               |
  WRITE/REPLACE   |                               |
                  |── TCP connect ───────────────>|
                  |── "2PC PRECOMMIT\n" ─────────>|  slave checks availability
                  |<─ "2PC ACK OK\n" ────────────|
                  |                               |
   (repeat for every peer)                        |
                  |                               |
  [Phase 2]       |── "2PC COMMIT WRITE …\n" ───>|  slave performs operation
                  |                               |  locally
                  |  master also performs locally |
                  |<─ "2PC DONE OK\n" ───────────|
                  |                               |
  (all DONE OK)   |── "2PC SUCCESS\n" ───────────>|  slave keeps result
                  |── TCP close ─────────────────>|
                  |                               |
  master replies "3.0 WROTE <id>" to client       |

5. ABORT PATHS
--------------

5a. Slave not available (cannot connect, or sends "2PC ACK FAIL"):

  MASTER                          SLAVE(s) already acked
    |── "2PC ABORT\n" ───────────>|  slave discards pending state
    |── TCP close ────────────────>|
    |
  master replies "3.2 ERROR" to client; bulletin board unchanged.

5b. Slave operation fails ("2PC DONE FAIL" from any slave):

  MASTER                          ALL SLAVES
    |── "2PC UNSUCCESS\n" ────────>|  slave rolls back (undo write/replace)
    |── TCP close ─────────────────>|
    |
  master rolls back its own local operation.
  master replies "3.2 ERROR" to client; bulletin board unchanged.

5c. Timeout waiting for slave response:

  If the master does not receive a response within TIMEOUT_SEC seconds
  (default 5 s) at any point, it treats the missing response as
  "2PC ACK FAIL" (in Phase 1) or "2PC DONE FAIL" (in Phase 2) and
  follows the abort / unsuccessful path above.

  If the slave does not receive "2PC SUCCESS" or "2PC UNSUCCESS" within
  TIMEOUT_SEC seconds after sending "2PC DONE OK", it rolls back
  (conservative safe choice).

6. FINITE AUTOMATA
------------------

MASTER automaton
~~~~~~~~~~~~~~~~
States: IDLE, PRECOMMIT_SENT, COMMITTED, DONE

  IDLE
    on WRITE/REPLACE from client
      → connect to all peers; send "2PC PRECOMMIT" to each
      → go to PRECOMMIT_SENT

  PRECOMMIT_SENT
    if any peer returns "2PC ACK FAIL" or times out
      → send "2PC ABORT" to all responding peers; close connections
      → reply "3.2 ERROR" to client
      → go to IDLE
    if all peers return "2PC ACK OK"
      → send "2PC COMMIT <op>" to all peers
      → perform operation locally
      → go to COMMITTED

  COMMITTED
    collect "2PC DONE OK/FAIL" from each peer (with timeout)
    if all "2PC DONE OK" AND local operation succeeded
      → send "2PC SUCCESS" to all peers; close connections
      → reply "3.0 WROTE/REPLACED" to client
      → go to IDLE
    else
      → send "2PC UNSUCCESS" to all peers; close connections
      → roll back local operation
      → reply "3.2 ERROR" to client
      → go to IDLE

SLAVE automaton
~~~~~~~~~~~~~~~
States: IDLE, PRECOMMIT_ACKED, COMMITTED

  IDLE
    on incoming TCP connection from master
      → receive "2PC PRECOMMIT"
      if currently busy (already in a 2PC transaction)
        → send "2PC ACK FAIL"; close; stay IDLE
      else
        → send "2PC ACK OK"; acquire busy lock
        → go to PRECOMMIT_ACKED

  PRECOMMIT_ACKED
    receive next message (with timeout)
    if "2PC ABORT" or timeout
      → release busy lock
      → go to IDLE
    if "2PC COMMIT WRITE <poster> <msg>"
      → attempt bbWrite(poster, msg)
      → send "2PC DONE OK" or "2PC DONE FAIL"
      → go to COMMITTED
    if "2PC COMMIT REPLACE <id> <poster> <msg>"
      → save original record for possible rollback
      → attempt bbReplace(id, poster, msg)
      → send "2PC DONE OK" or "2PC DONE FAIL"
      → go to COMMITTED

  COMMITTED
    receive next message (with timeout)
    if "2PC SUCCESS" or timeout after sending DONE FAIL (no-op rollback needed)
      → keep result; release busy lock
      → close connection; go to IDLE
    if "2PC UNSUCCESS" or timeout after sending DONE OK
      → undo the local operation (remove written record, or restore replaced record)
      → release busy lock
      → close connection; go to IDLE

7. CONCURRENCY NOTES
--------------------
- Only one 2PC transaction runs on a slave at a time (g_slaveBusy flag).
  A slave that is already participating in a transaction replies
  "2PC ACK FAIL" to any new PRECOMMIT it receives.

- Client READ requests are never blocked by an ongoing 2PC on the same
  server (the readers-writer lock allows concurrent reads).

- Client WRITE/REPLACE requests that arrive at a slave while it is
  participating in a 2PC as a slave will themselves become masters and
  will send PRECOMMIT to their own peers.  The peer that is currently
  acting as slave for another transaction will reply "2PC ACK FAIL",
  causing the second transaction to abort.  This is intentional:
  consistency is preserved at the cost of availability.

8. EXAMPLE SESSION TRANSCRIPT (pdebug output)
---------------------------------------------

  Client → Master:   WRITE alice Hello world
  Master → SlaveA:   2PC PRECOMMIT
  SlaveA → Master:   2PC ACK OK
  Master → SlaveA:   2PC COMMIT WRITE alice Hello world
  (Master writes locally: id = 42)
  SlaveA → Master:   2PC DONE OK
  Master → SlaveA:   2PC SUCCESS
  Master → Client:   3.0 WROTE 42

  --- failure example ---
  Client → Master:   WRITE bob Test
  Master → SlaveA:   2PC PRECOMMIT
  (SlaveA is busy)
  SlaveA → Master:   2PC ACK FAIL
  Master → Client:   3.2 ERROR WRITE replication failed

9. DESIGN RATIONALE
-------------------
- Plain text was chosen for debuggability; the protocol can be tested
  with telnet or nc on RPORT.
- The "2PC " prefix on every message unambiguously separates replication
  traffic from the client application protocol.
- Each phase has an independent per-message timeout (TIMEOUT_SEC = 5 s)
  rather than a single end-to-end timer, so a slow but functional slave
  does not unfairly abort fast transactions.
- The slave rolls back on timeout after phase 2 (conservative choice) to
  avoid leaving a committed record that the master thinks was rolled back.
